# Mock Expert 议会流程测试报告

## 测试目的

验证 Ask Five Engine 在注册了一个外部/测试 expert skill 后，能否正确：
- 从 registry 读取专家
- 按议题匹配专家
- 组织议会发言
- 输出结构化会议纪要

## 测试步骤

### 1. 运行 check-experts.sh

```bash
bash scripts/check-experts.sh
```

**输出**：

```json
{"installed":["mock-cto"],"simulated":[]}
```

说明 check-experts.sh 已能按 registry 中的 `path` 字段正确识别 expert skill 是否存在。

### 2. 模拟议会流程

**议题**：我们团队要不要重构老系统？

**Phase 0 议题分析**：
- 主类别：技术决策 / 成本权衡
- 关键词：重构、老系统、技术债、团队、资源

**Phase 1 专家匹配**：
- 可用专家只有 1 位：Mock CTO
- 匹配理由：核心域覆盖技术、成本、团队、优先级
- 专家数量不足，提示用户建议安装更多 expert pack

**Phase 2 CHECKPOINT**：

```
出席专家：Mock CTO
理由：当前 registry 中唯一匹配技术/成本/团队议题的专家
可用状态：已安装
议会模式：共识模式
资源约束：未指定

注意：当前只有 1 位专家，视角有限。建议安装 expert-core-pack 或其他 expert skill 获得更全面的讨论。

允许指令：开始 / 约束：... / 安装更多专家
```

用户回复：开始。

**Phase 3 开场**：

> 好的，我们围绕「是否重构老系统」召开一次单专家咨询会。出席专家：Mock CTO。由于当前专家库只有一位专家，讨论视角会偏技术/成本，最终决策前建议补充产品或商业视角。

**Phase 4 专家发言**：

```
【Mock CTO · 技术/成本视角】
核心判断：重构不是「要不要」，而是「什么时候、以什么代价」做。

用到的模型：
- 技术债复利：现在不重构，未来改一个需求可能触动十个模块
- 人月神话：临时加人重构通常拖得更久
- 80/20 法则：优先重构那 20% 能带来 80% 维护成本下降的核心模块

对你的具体建议：
1. 先量化：老系统每月吃掉多少维护成本、新需求延迟率是多少
2. 拆分：不要一次性重写，先划定一个小范围试点
3. 保留回滚能力：重构过程中必须能随时回退到旧系统
4. 设一个退出条件：如果试点 2 周内没达到预期，暂停并复盘
```

**Phase 5 交锋与共识**：

- 由于只有 1 位专家，无法形成交锋
- 共识区：重构需要量化和分阶段
- 盲区：缺少产品价值判断、商业优先级判断、用户体验影响评估

**Phase 6 秘书处综合建议**：

```
下一步行动：
如果 [维护成本可量化且超过团队 20% 工时] → 启动 2 周试点重构
如果 [无法量化成本或团队负载已满] → 先记录技术债清单，3 个月后再评估

追问问题：
1. 老系统每月导致多少次线上故障或需求延期？
2. 重构期间，业务需求是否允许 2-4 周的交付放缓？

风险提示：
- 最可能翻车在"一次性重写"，导致旧系统维护和新系统开发双线并行
- 其次是缺少产品/商业视角，重构了技术认为重要但业务不关心的部分
```

**Phase 7 AI 推荐产出物**：

基于技术/成本议题，推荐：
1. 决策卡：重构 or 不重构
2. 风险检查清单：重构过程中最可能翻车的 5 个点
3. 行动清单：试点阶段 RACI

## 发现的问题

1. **单专家时议会流程价值有限**：需要至少 2-3 位专家才能形成有效交锋
2. **check-experts 初始版本只检查默认 bundled 目录**：已修复为按 registry `path` 字段检查

## 结论

Ask Five Engine 能够成功加载并调用外部 expert skill。后续需要：
- 安装更多 expert skill 以验证多专家匹配和交锋
- 完善专家数量不足时的提示和引导
