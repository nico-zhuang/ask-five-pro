# Brain Crew 专家准入标准

本文档定义什么样的「人」可以进入 Brain Crew，以及什么样的 skill 质量可以被接受。

---

## 1. 人选标准

### 1.1 入选条件

Brain Crew 只收录真实存在、有公开言论/作品/决策记录可考的人物，或已被验证有效的方法论化身。

**优先收录**：

- 有广泛公开的一手材料（演讲、文章、书籍、访谈、决策记录）
- 在某一领域有长期、可验证的杰出成果
- 思维方式独特且可提炼为可操作框架
- 对中文语境下的产品、商业、创作有实际参考价值

**避免收录**：

- 虚构人物或争议过大、缺乏一手材料的人物
- 以冒犯或煽动为核心标签的人物
- 与现有专家严重重叠的人物

### 1.2 来源要求

- 优先使用公开可验证的来源
- 不允许编造引文、言论、传记细节
- 对来源的推断必须标注为推断，不能包装成事实

---

## 2. Skill 质量标准

### 2.1 Frontmatter 规范

每个专家 `SKILL.md` 必须包含以下字段：

```yaml
id: expert-id                 # 必填，kebab-case，全局唯一
name: 专家姓名                 # 必填
pack: core                    # 必填，见 CONTRIBUTING.md
description: >                 # 必填，≤1024 字符
  一句话说明：这个专家是谁、何时触发、核心用途。
core_domain: ["领域1"]        # 必填，2-4 个字符串
style: ["风格1"]              # 必填，2-4 个字符串
topic_tags: ["标签1"]          # 必填，5-8 个字符串
sensitivity: normal           # 必填，normal / sensitive
version: 1.0.0                # 必填，semver
```

### 2.2 内容结构要求

| 模块 | 要求 | 目的 |
|------|------|------|
| 角色扮演规则 | 明确「我」视角、退出触发、免责声明次数 | 防止破角色 |
| 触发场景 | 列出典型问题类型和关键词 | 便于 Query API 匹配 |
| 工作流 Step 1/2/3 | 明确的 Agentic Protocol | 保证输出一致性 |
| CHECKPOINT | ≥1 个关键节点自检 | 防止硬答、越界 |
| 心智模型 | 1-5 个核心模型/原则 | 提供可调用框架 |
| 失败模式表 | if-then 结构，含症状和应对措施 | 提升鲁棒性 |
| 反例黑名单 | ≥5 条常见误用及正确做法 | 防止错误模仿 |
| 应答示例 | 1-2 个具体 Q&A，符合专家 voice | 提升可执行性 |

### 2.3 表达质量要求

- **减少软化措辞**：避免大量「建议」「可以考虑」「视情况而定」。改为直接判断或条件分支。
- **数字锚点**：关键建议应包含时间、预算、人数、指标等可量化锚点。
- **专家 voice**：输出应符合该人物的表达 DNA，不要为了结构化而变得机械。
- **反 AI 味**：避免堆砌特征、listicle、过度总结、空泛鼓励。

## 3. Darwin 评分维度

Brain Crew 使用简化 Darwin 9 维评分，权重如下：

| 维度 | 权重 | 考察点 |
|------|------|--------|
| Frontmatter 质量 | 7% | 字段完整、description 清晰 |
| 工作流清晰度 | 12% | 明确 Step 1/2/3、`## 回答工作流` 标题 |
| 失败模式编码 | 12% | 有失败模式表、含 if-then 分支 |
| 检查点设计 | 6% | CHECKPOINT 数量 ≥1，建议 ≥2 |
| 可执行具体性 | 17% | 少软化词、有数字锚点、输出可直接执行 |
| 资源整合度 | 4% | references/、scripts/、assets/ 等 |
| 整体架构 | 12% | 标题清晰、无冗余 AI 腔、h2 不过度 |
| 实测表现 | 23% | 需人工跑测试 prompt，默认给基准分 7 |
| 反例与黑名单 | 6% | 有反例表、含 why-wrong |

### 3.1 准入分数线

- **单个专家**：≥ 85/100
- **全量平均**：≥ 86/100
- **优秀线**：≥ 90/100（鼓励 but 不强制）

---

## 4. 敏感专家额外要求

敏感专家（`sensitivity: sensitive`）需满足：

- 必须放在 `enterprise` pack
- 必须有明确的角色边界和免责声明
- 必须标注「可能引发争议」的典型场景
- 默认不被 `query_experts()` 匹配，仅在 `mode=debate` 启用
- 维护者需额外审核其政治、伦理风险

---

## 5. 迭代与淘汰

- 每次版本更新后必须重跑 Darwin 评分
- 连续两个版本评分低于 85 的专家进入观察期
- 如果无法通过内容优化回到 85，考虑移出核心包或降级为扩展包

---

## 6. 快速自检问题

新增专家前，问自己：

1. 这个人有公开、可验证的一手材料吗？
2. 他的思维方式能被提炼成 1-3 个核心模型吗？
3. 用户什么情况下会特别想调用他？
4. 他的回答会不会和现有某位专家严重重叠？
5. 他的失败模式有哪些是容易犯的、可以提前编码的？
6. 跑完 Darwin 评分后，有没有维度低于 7 分？

如果 6 个问题都能给出肯定且具体的答案，这位专家适合进入 Brain Crew。
