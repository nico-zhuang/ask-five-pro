# Ask Five Engine · 真实会议测试

**议题**：Brain Crew 从 25 人扩充到 30 人后，下一步该怎么优化？

**会议模式**：共识模式

**出席专家**：
- 达尔文（meta · skill 优化）
- 史蒂夫·乔布斯（core · 产品聚焦）
- 张一鸣（core · 系统设计）
- 杰夫·贝佐斯（enterprise · 运营平台化）
- 黄仁勋（tech · 生态基础设施）

**选派理由**：这是一个关于 skill 产品如何持续进化的 meta 议题，需要质量评估、产品聚焦、系统架构、运营平台和生态思维五个视角。

---

## 专家发言

### 达尔文 · skill 优化视角

**核心判断**：人数从 25 到 30 只是量变，真正的优化应该建立**可量化的质量反馈回路**。

**具体建议**：
1. 为每个专家设计 2-3 个标准测试 prompt，定期跑分。
2. 记录每个专家在实际会议中的"采纳率"——用户听了建议后是否行动。
3. 把评分低于 75 的专家移入观察池，不直接删除，给 2 周改进期。
4. 每增加一个新专家，必须同时更新 `test-prompts.json` 和 `decisions-log.md`。

### 史蒂夫·乔布斯 · 产品聚焦视角

**核心判断**：30 个专家已经太多了。**你不是在做一个百科全书，你是在做一个能被调用的判断系统。**

**具体建议**：
1. 问一个残酷问题："如果只能留 10 个专家，谁留下？"
2. 把专家按"不可替代性"排序，重叠视角的合并或降级到扩展包。
3. Core Pack 必须保持 6-8 人，超过这个数，engine 的选派精度会下降。
4. 与其加更多专家，不如让现有 30 个的输出质量翻倍。

### 张一鸣 · 系统设计视角

**核心判断**：优化的关键不是专家数量，而是**匹配算法的信号质量**。

**具体建议**：
1. 现在 `topic_tags` 和 `core_domain` 有重叠，应该合并或分层：
   - `core_domain`：专家能回答什么大类问题
   - `topic_tags`：具体触发词
   - `style`：议会模式偏好（对抗/共识/红队）
2. 引入用户反馈闭环：每次会议结束后，让用户标注"哪个专家最有用"。
3. 用反馈数据训练一个简单的匹配权重，而不是靠人工调分。

### 杰夫·贝佐斯 · 运营平台化视角

**核心判断**：Brain Crew 应该从一个 pack 进化为一个**可被外部贡献的平台**。

**具体建议**：
1. 定义清晰的"专家入驻标准"：frontmatter 规范、测试 prompt、最低评分、许可证声明。
2. 把贡献流程文档化：任何人都可以 PR 一个新 expert，但必须先通过 `check-registry.sh` 和 `darwin-quick-score.py`。
3. 把 core/brand/tech/enterprise/meta 看作不同的"产品线"，各自有 owner 和质量负责人。
4. 长期目标：Brain Crew 成为 Ask Five 生态的"默认应用商店"。

### 黄仁勋 · 生态基础设施视角

**核心判断**：不要把 Brain Crew 当内容库，要把它当**基础设施层**。

**具体建议**：
1. 专家不是 SKU，是 API。每个专家应该能被 engine 稳定调用，输出格式一致。
2. 重点投资"接口层"：统一的 frontmatter、统一的输出结构、统一的失败 fallback。
3. 当接口稳定后，专家数量自然会增长，因为外部贡献成本降低了。
4. 现在最该做的事：把 registry 和 engine 的契约写死，未来 5 年不变。

## 共识 / 分歧 / 盲区

**共识**：
- 30 人不是终点，质量反馈回路和入驻标准比继续加人更重要。
- Core Pack 必须保持小而精。
- registry 与 engine 的契约需要长期稳定。

**分歧**：
- 达尔文想先建评分系统再优化；乔布斯想先做减法；贝佐斯想做平台化。三者不冲突，但优先级不同。
- 张一鸣强调算法和数据闭环，黄仁勋强调接口稳定性，两者都是基础设施但侧重点不同。

**盲区**：
- 没有讨论用户实际使用数据。优化的依据应该是真实反馈，而不是理论推演。
- 没有讨论企业私有 expert 的接入机制。
- 没有讨论敏感专家（Trump/张雪峰）是否真的需要保留在 enterprise pack。

## 秘书处综合建议

### 下一步行动（按优先级）

1. **P0：建立质量反馈回路**
   - 为 30 位专家各写 2 个测试 prompt
   - 每月跑一次 `darwin-quick-score.py`
   - 低于 75 分的专家进入"观察池"

2. **P1：精简 Core Pack 候选**
   - 评估是否把贝佐斯或黄仁勋加入 Core Pack
   - 如果加入， Core Pack 变为 7-8 人；如果不加入，明确 core 的入选门槛

3. **P2：写贡献者文档**
   - `CONTRIBUTING.md`：如何提交新 expert
   - `EXPERT_CRITERIA.md`：入驻标准、评分门槛、许可证要求

4. **P3：收集真实用户反馈**
   - 在 Ask Five 输出后加一个评分入口
   - 记录"哪位专家最有用"，用于优化匹配算法

### 追问问题

1. Brain Crew 的目标用户是谁？个人决策者、企业团队、还是开发者？
2. 你更在意"覆盖面广"还是"核心场景输出质量高"？
3. 是否允许外部贡献者提交 expert？还是只由内部维护？

### 风险提示

- 继续快速扩充专家而不建质量门槛，会导致 engine 选派准确率下降。
- registry 和 engine 的契约如果不稳定，未来每次扩展都会牵一发而动全身。
- 缺乏用户反馈数据时，所有"优化"都是基于假设。
