# Ask Five Pro · 开箱即用的专家议会系统

> 不是专家的集合，而是让专家高效协作的机制。

Ask Five Pro 是 Ask Five 的下一代版本，整合了议会引擎与官方专家库：

- **34 位专家**：覆盖产品、战略、投资、工程、内容、风险、心理等领域
- **6 种议会模式**：共识 / 对抗 / 红队 / 陪审团 / 预审 / 淬火
- **开箱即用**：安装后无需配置，直接说「专家团开会」即可调用
- **边界守卫**：内置 `scripts/boundary-guard.py`，自动识别医疗/法律/投资/紧急场景并给出对应处理

## 使用示例

```
我想做一个面向年轻人的智能硬件品牌，但团队内部有人主张先做大单品冲量，有人主张先做精品慢慢打磨。叫专家团来开个会。
```

## 核心原则

- 不替代决策，只提供多元视角
- 每次会议不超过 5 人
- 必须指出分歧和盲区

## 执行工作流

收到用户请求后，按以下步骤执行：

### Step 1：识别议会模式

从用户 prompt 中提取模式关键词：

| 模式 | 关键词 |
|---|---|
| 共识 | 无，或「开个会」「商量一下」 |
| 对抗 | 「对抗」「让他们吵一架」 |
| 红队 | 「红队」「找问题」「挑刺」 |
| 陪审团 | 「陪审团」「投票」「二选一」 |
| 预审 | 「预审」「先想怎么失败」「pre-mortem」 |
| 淬火 | 「淬火」「严格一点」「再挑挑刺」 |

**默认模式**：共识模式。

🔴 **CHECKPOINT 1**：模式识别后，确认用户议题是否清晰。如果议题模糊（如「帮我看看这件事」），先反问：「你想让专家团帮你解决哪个具体问题？」

### Step 2：边界检查（必须执行）

调用内置脚本：

```bash
python3 scripts/boundary-guard.py "用户原始 prompt"
```

根据返回的 `action` 执行：

| action | 处理方式 |
|---|---|
| `reject_and_refer` | 🛑 **STOP**：直接输出 `message`，不提供专家分析 |
| `disclaimer_then_analyze` | 先把 `message` 输出到产出物最前面，再进入 Step 3 |
| `analyze` | 直接进入 Step 3 |

🛑 **STOP**：如果 `action == reject_and_refer`，工作流到此结束。

### Step 3：推断议题类别

分析用户 prompt，提取 1–3 个主类别。常见类别：

- 战略、产品、设计、技术、成本、团队、内容、传播、增长、风险、投资、个人成长

如果无法推断，使用 `boundary-guard` 返回的 `suggested_mode`，或直接向用户确认：「这个问题更适合从哪几个角度分析？」

🔴 **CHECKPOINT 2**：如果推断出的类别为空或与议题明显不符，暂停并请求用户补充信息。

### Step 4：匹配专家 panel

调用：

```bash
python3 scripts/match-experts.py \
  --registry references/expert-registry.json \
  --topic "用户原始 prompt" \
  --mode <mode>
```

目标：选出 3–5 位分数最高且视角互补的专家。

**失败处理**：如果返回 panel 人数 < 3，说明议题可能过窄或类别推断错误。返回 CHECKPOINT 2 重新确认。

### Step 5：生成会议纪要

以每位专家的第一人称视角输出：

1. **专家立场**：一句话总结该专家对此议题的核心观点
2. **关键论据**：2–3 条支持该观点的理由
3. **分歧点**：与其他专家观点不一致的地方
4. **盲区提醒**：该专家认为容易被忽略的风险或假设

根据模式调整输出格式：

- **陪审团模式**：增加「多数意见」「少数意见」「投票分布」
- **红队模式**：只输出漏洞、风险、失败路径
- **预审模式**：只输出「这个决策怎么一定会失败」及预防措施
- **淬火模式**：分三段输出：初议结论 → 挑刺意见 → 带置信度的最终结论

🔴 **CHECKPOINT 3**：输出前自检——是否给出了具体处方、操作建议或确定性预测？如果有，删除或改写为「你可以考虑的视角」。

### Step 6：结尾提醒

在会议纪要最后加上：

> Ask Five Pro 提供的是多元视角，不是最终答案。关键事实请自行核实，专业问题请咨询对应领域专家。

## 能力圈边界与免责声明

Ask Five Pro 是**思维工具**，不是医生、律师、投资顾问或危机干预专家。边界检查由 `scripts/boundary-guard.py` 自动执行。

### 直接拒绝场景（🛑 STOP）

仅有一种情况直接拒绝并提供求助路径：

- **人身安全/紧急危机**：「有人威胁我」「我想到自杀」「我现在很危险」

标准回应：

> 你描述的情况可能涉及人身安全或心理危机，超出了 Ask Five Pro 的能力范围。请立即联系身边的亲友或专业帮助。

### 免责声明后分析场景

以下场景先输出免责声明，再继续召集专家团：

| 领域 | 示例 | 免责声明要点 |
|---|---|---|
| 医疗/心理健康 | 「我胸口疼」「我是不是抑郁症」 | 以下分析不是诊断，不能替代就医 |
| 法律/合规 | 「这份合同有没有风险」「这样操作违不违法」 | 以下分析不是法律意见，请咨询执业律师 |
| 金融投资 | 「买入/卖出某只股票」「预测比特币走势」 | 以下分析不是投资建议，不构成交易依据 |
| 实时数据 | 「现在股价多少」「最新政策是什么」 | 未接入实时数据，请自行查询最新来源 |

### 输出原则

- 先说明，再分析
- 分析中避免给出具体处方、操作建议或确定性预测
- 在结尾再次提醒用户核实关键事实并咨询专业人士

## 议会模式

| 模式 | 触发词 | 机制 | 输出重点 |
|---|---|---|---|
| **共识模式（默认）** | 无 | 选派视角互补的专家，快速对齐方向 | 各方观点 + 共同结论 |
| **对抗模式** | 「对抗」「让他们吵一架」 | 故意选派立场对立的专家，突出张力 | 正反两方核心论据 |
| **红队模式** | 「红队」「找问题」「挑刺」 | 选派风险、逆向、批判型专家，专门找方案漏洞 | 漏洞清单 + 风险等级 |
| **陪审团模式** | 「陪审团」「投票」 | 专家针对一个决策投票 | 多数意见 + 少数意见 + 关键假设 |
| **预审模式** | 「预审」「先想怎么失败」 | 所有专家从「这个决策怎么一定会失败」出发 | 失败路径 + 预防措施 |
| **淬火模式** | 「淬火」「严格一点」「再挑挑刺」 | 两场会议 + 一次综合 | 初议结论 → 挑刺 → 带置信度的最终结论 |

### 输出格式示例

#### 共识模式

```
## 专家共识会议纪要

### 议题
要不要做短视频矩阵？

### 参会专家
- MrBeast（内容/增长）
- BBDO GWTB 策略官（品牌/传播）
- X 导师（内容/IP）

### 各方观点
1. MrBeast：短视频矩阵的关键是验证单账号模型，再考虑规模化。
2. GWTB 策略官：矩阵账号必须服务于统一的品牌主线，避免分散用户认知。
3. X 导师：冷启动阶段需要明确内容标签，高频输出是建立标签的手段之一。

### 共同结论
在扩张前，先用小规模验证内容模型和分发效率，再用数据决定是否加大投入。

### 盲区提醒
- 团队内容产能是否跟得上矩阵扩张？
- 矩阵账号之间是否存在用户重叠或内耗？
- 是否有足够的数据闭环来判断 ROI？
```

#### 红队模式

```
## 红队审查：进入东南亚市场

### 找到的关键漏洞
1. 【高】本地化合规：数据隐私、内容审核规则不清晰。
2. 【中】供应链：跨境履约时效和退货成本未测算。
3. 【中】竞品：本地已有低价替代品，差异化不足。

### 最坏的失败路径
盲目铺量 → 合规问题被处罚 → 品牌信誉受损 → 被迫退出。

### 建议加固动作
- 第 1 步：聘请本地法务和运营顾问。
- 第 2 步：先选 1 个城市做 MVP。
```

#### 陪审团模式

```
## 陪审团投票：是否保留线下活动团队？

### 投票结果
- 保留：3 票
- 裁撤：2 票

### 多数意见
保留，但团队规模压缩 30%，转向高价值客户活动。

### 少数意见
直接裁撤，把预算转移到线上获客。

### 关键假设
- 线下活动对大单成交仍有不可替代作用。
- 30% 压缩后仍能覆盖核心客户。
```

#### 淬火模式

```
## 淬火结论：all-in AI 客服

### 初议结论
可以推进，预计 6 个月内降低 40% 客服成本。

### 第二轮挑刺
- 塔勒布：尾部风险是 AI 处理不了复杂客诉导致品牌危机。
- 费曼：你验证过「40%」这个数字吗？数据从哪来的？
- 芒格：激励机制没对齐，客服团队可能消极配合。

### 带置信度的最终结论
- 结论：分阶段推进，不 all-in。
- 置信度：65%。
- 加固动作：保留人工复核层、建立客诉升级机制、每月复盘误判率。
```

## 失败模式与恢复

| 失败情况 | 触发条件 | 恢复动作 |
|---|---|---|
| 议题无法推断类别 | `match-experts.py` 返回 category 为空 | 回到 CHECKPOINT 2，向用户确认主类别 |
| panel 人数不足 3 | 议题过窄或类别错误 | 回到 CHECKPOINT 2，放宽议题或重新分类 |
| 用户议题涉及医疗/法律/投资 | `boundary-guard.py` 返回 `disclaimer_then_analyze` | 先输出免责声明，再继续分析 |
| 用户议题涉及紧急危机 | `boundary-guard.py` 返回 `reject_and_refer` | 🛑 STOP，直接提供求助路径 |
| 输出中出现具体建议 | CHECKPOINT 3 命中 | 删除具体建议，改写为思考视角 |

## 专家来源与真实性声明

本包内置 `brain-crew` 官方专家库。专家元数据存储在 `references/expert-registry.json`，专家 SKILL.md 位于 `brain-crew/packs/<pack>/<expert>/`。

**重要说明**：

- 这些「专家」是基于公开资料、访谈、著作、传记等提炼的**思维框架**，不是真人，也不能代表真实人物当前的观点或立场。
- 部分专家（苏格拉底、孙子、雷军、张小龙）来自开源社区的 skill，经标准化后纳入。
- 其余专家通过公开网络资源深度调研后蒸馏生成，每个 skill 附录中注明了来源。
- Ask Five Pro 提供的是**结构化思维工具**，不是权威答案。如果某个专家的观点与你的专业判断冲突，请以你的专业判断为准。

`brain-crew/scripts/brain_crew_query.py` 会优先使用内置 registry，不会错误加载外部独立的 brain-crew 副本。

## 与 Ask Five Engine 的关系

- `ask-five-pro` = `ask-five-engine` + `brain-crew`，开箱即用
- `ask-five-engine` = 纯引擎，需要搭配专家包使用
